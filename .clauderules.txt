## ⚫ RÈGLE 15 — BOUCLIER ANTI-RACE DES STUBS `_mzGuard`
**Le système `_mzGuard` (`index.html:28-82`) installe des stubs vides pour toutes les fonctions appelées en `onclick`.**
- Règle : `_mzGuard` ne définit la fonction QUE si `typeof window[name] !== 'function'`
- Conséquence : **toute fonction définie dans un script `.js` doit être EXPLICITEMENT ré-assignée à `window.name`** — sinon `_mzGuard` gagne et le bouton est muet (cf. commit `3223779`)
- Checklist obligatoire : pour chaque fonction listée dans `_mzGuard`, vérifier la présence d'un `window.xxx = xxx;` en fin de `engine.js` ou `app.js`
- Interdit : ajouter un nouveau stub `_mzGuard` sans créer l'implémentation réelle dans la même PR

La précision prime sur la volumétrie. Un résultat fiable > dix résultats douteux.
- Le paramètre `limit` ou `max_results` dans les appels API doit être codé en dur à `1` dans la couche `services/`
- Vercel a un timeout strict : chercher plus est un luxe qui tue la requête
```python
# MAX_RESULTS = 1 : Stratégie Sniper — évite les doublons et les timeouts
# cascade sur l'API Dorar. Ne pas modifier sans test de charge.
MAX_RESULTS = 1
```

### 🔴 LOI V3-3 — CONTRAT DE RÉPONSE IMMUABLE
Chaque endpoint Vercel retourne TOUJOURS ce schéma, succès ou erreur :
```json
{ "status": "ok|error", "data": {}, "error_code": null, "duration_ms": 0 }
```
Ne jamais retourner une exception brute ou un HTML 500 au frontend.

### LOI V3-4 — IDEMPOTENCE DES REQUÊTES
Tout appel API externe doit pouvoir être rejoué sans effet de bord.
Implémenter un `request_id` unique côté client, loggé côté serveur.
- Backend : Générer un `X-Request-ID` unique pour chaque requête
- Frontend : `if (processedIds.has(data.id)) return;` avant tout rendu DOM

## ⏱️  BLOC II — TIMEOUTS & RÉSILIENCE RÉSEAU

### 🔴 LOI V3-5 — LA RÈGLE DES 3 NIVEAUX DE TIMEOUT
**PRIORITÉ SURVIE — Ces valeurs sont des CONSTANTES nommées, jamais des magic numbers.**
```
Niveau 1 — httpx/requests timeout    :  8s  (appel Dorar/HadeethEnc)
Niveau 2 — Vercel function timeout   : 25s  (limite plan Hobby = 30s)
Niveau 3 — Frontend fetch timeout    : 20s  (AbortController JS)
```
Envelopper les appels `httpx` dans `asyncio.wait_for(..., timeout=8.0)`.
Configurer `maxDuration: 10` dans `vercel.json`.

### LOI V3-6 — ZÉRO RETRY AVEUGLE
Pas de retry automatique sans backoff exponentiel ET vérification
que l'erreur est bien retriable (5xx, timeout) et non définitive (4xx).
- Maximum : 2 retries. Délais : 1s, 3s
- Circuit breaker : après 5 échecs consécutifs sur 30s → fallback + alerte

### 🔴 LOI V3-7 — KEEP-ALIVE EXPLICITE POUR SSE (Mode Sniper Séquentiel)
**PRIORITÉ SURVIE — Ce mécanisme ne s'adapte pas "au contexte" : il est TOUJOURS actif.**
- Le ping SSE `data: ping\n\n` s'envoie toutes les **15s** maximum (côté serveur)
- Si le client ne reçoit rien pendant 25s → reconnexion automatique côté JS
- Pas de `gather` asyncio pour les pings SSE — traitement **séquentiel** obligatoire
```javascript
// Vanilla JS Obligatoire — Détection timeout SSE
let lastPing = Date.now();
source.addEventListener('ping', () => lastPing = Date.now());
setInterval(() => { if (Date.now() - lastPing > 30000) reconnect(); }, 5000);
```
- Reconnexion automatique avec backoff (1s → 3s → 8s) + `visibilitychange` + `online/offline`
- Jamais de `setInterval` pour polling ; uniquement SSE + fallback HTTP long-polling si SSE échoue 3 fois

### LOI V3-8 — GRACEFUL DEGRADATION OBLIGATOIRE
Si Dorar est inaccessible, retourner un message d'erreur structuré
(avec `error_code: "SOURCE_UNAVAILABLE"`), jamais un résultat vide silencieux.
L'utilisateur doit TOUJOURS savoir pourquoi il n'a pas de résultat.
Jamais de Spinner Infini — si le stream casse : "Connexion interrompue - Nouvelle tentative dans 3s..."

## ⚙️  BLOC III — SCOPE CLIENT HTTP & INDENTATION PYTHON

### 🔴 LOI V3-9 — SCOPE DU CLIENT HTTP (Indentation Python)
**PRIORITÉ SURVIE — Faille critique identifiée sur Vercel Serverless.**
Le client `httpx` (ou `requests`) NE DOIT PAS être instancié au niveau module (scope global).
Il DOIT être instancié à l'intérieur du scope de la fonction handler :
```python
# ❌ INTERDIT — scope module (partagé entre invocations, provoque des fuites)
client = httpx.AsyncClient(timeout=8.0)

async def handler(request):
    response = await client.get(url)

# ✅ OBLIGATOIRE — scope fonction (isolé par invocation Vercel)
async def handler(request):
    async with httpx.AsyncClient(timeout=8.0) as client:
        response = await client.get(url)
```
- Aucun state partagé entre invocations Vercel (stateless only)
- Typing strict (`from __future__ import annotations`), pas de `Any`, pas de `global`

### LOI V3-10 — MODE SNIPER SÉQUENTIEL (Pas de gather)
**PRIORITÉ SURVIE — asyncio.gather interdit sur les appels Dorar.**
Les appels à l'API Dorar doivent être exécutés **séquentiellement**, pas en parallèle.
- `asyncio.gather()` est interdit pour les requêtes vers Dorar.net
- Raison : MAX_RESULTS=1 + séquentialité = zéro doublon, zéro race condition, zéro timeout cascade
- Si plusieurs appels sont nécessaires → boucle `for` avec `await` séquentiel

## 🔓  BLOC IV — DÉBLOCAGE UI (Index 0)

### 🔴 LOI V3-11 — AUTORISER L'INDEX 0 DANS LES RÉSULTATS
**PRIORITÉ SURVIE — Bug silencieux identifié : l'index 0 est un résultat valide.**
Toute vérification de résultat ou d'index NE DOIT PAS utiliser un test de vérité implicite sur 0 :
```javascript
// ❌ INTERDIT — bloque l'affichage du premier résultat (index 0 = falsy)
if (result.index) { render(result); }
if (data.hadith_id) { display(data); }

// ✅ OBLIGATOIRE — vérification explicite
if (result.index !== undefined && result.index !== null) { render(result); }
if (data.hadith_id !== undefined && data.hadith_id !== null) { display(data); }
```
```python
# ❌ INTERDIT en Python aussi
if result_index:
    process(result_index)

# ✅ OBLIGATOIRE
if result_index is not None:
    process(result_index)
```
- S'applique à : `hadith_id`, `index`, `idx`, tout identifiant numérique pouvant valoir `0`
- Un `if (value)` sur un entier = bombe silencieuse qui tue l'affichage du premier résultat
